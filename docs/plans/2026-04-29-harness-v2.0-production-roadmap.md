## North star

> **Pick one core job and do it exceptionally well.**

The core job: **make AI-assisted development inside any consumer repo not drift, without burning tokens, without breaking trust.**

Everything in v2.0 is in service of that. Anything that drags us away from it (a web UI, an LLM-judge tier, a sandbox, an evals platform) is explicitly out of scope. The HARNESS_CARD's `not_covered` field is treated as a contract.

## Operating principles for the whole sprint

These are non-negotiable. Every story is reviewed against them before it's marked done.

1. **Reliability over feature velocity.** No story merges until it has tests and a passing CI. A flaky test is a failing test.
2. **Maintainability over cleverness.** A 30-line check that any contributor can read beats a 10-line check with metaclasses.
3. **No silent error handling.** Every failure path either:
   - emits a clearly-formatted error the user can act on, OR
   - exits with a documented non-zero code that the orchestrator surfaces.
   Bare `except: pass` is a bug; `except Exception:` without a logger or re-raise is a bug. The harness already enforces this on consumers (Q17); v2.0 holds itself to the same standard.
4. **Clear contracts between components.** Every internal interface is documented (function signature + docstring + exact return shape). The H-16 emit format is the canonical inter-process contract. Two components never share state through "well, they both happen to use the same Path" — they share via explicit arguments or via a documented file contract.
5. **Test coverage gate.** ≥95% line + branch coverage on `tools/` and `.harness/checks/` measured per release. Cumulative test count grows; never shrinks without an ADR.
6. **Stack profiles, not stack assumptions.** Every check declares which profiles it belongs to. The orchestrator only runs checks whose profile is enabled in the consumer's `.harness/profile.yaml`.
7. **First-run delight, not first-run trauma.** A fresh `harness init` on any repo prints a single green-checkmark summary, not a wall of legacy violations.
8. **Permanent regression tests.** Every closed bug (B1–B27, S-A* through S-DP*) has a permanent test that re-fires if the regression returns. Tests are never deleted; they are renamed when superseded.
9. **No undocumented exit codes.** Every non-zero exit code is listed in `docs/EXIT_CODES.md` with its meaning and the recovery path.
10. **Cross-platform CI matrix.** Every CI job runs on `ubuntu-latest` AND `macos-latest`. Windows is explicitly **unsupported but documented** (see Sprint 0 S0.6).

## Sprint 0 — Foundations (1 week, blocks Sprint 1)

**Goal:** make the rest of v2.0 buildable with high confidence. This sprint is about architectural decisions, tooling, and a regression-test floor that prevents bugs from re-shipping.

**Why first:** Sprints 1+ depend on decisions that haven't been made yet (Node parser choice, severity tiers, detection signals). Trying to make those decisions during execution leads to half-finished prototypes and rework.

### Story v2.S0.1 — Node-pack architecture decision (ADR)

**Files:**
- `docs/decisions/2026-04-29-node-ast-parser.md`
- `tools/perf/node_parser_bench.py` (one-off benchmark; not shipped)

**The decision:** how do Node-pack checks parse JS/TS source?

**Options to benchmark:**
- **A.** `tree-sitter-javascript` + `tree-sitter-typescript` Python bindings. In-process, fast per-call (~10ms after warm), heavy install (compiled C extensions, ~30 MB).
- **B.** Subprocess to `node --eval` running esprima/babel. Zero install footprint when Node-pack disabled, slow per-call (~150ms cold start each), needs Node already installed.
- **C.** Subprocess to `swc` or `oxc` (Rust-native parsers). Fast even cold (~30ms), heavy install (Rust binaries).

**Acceptance:**
- Benchmark each option on a fixture corpus (50 small files + 5 large files).
- ADR documents the choice + reasoning + the rejected alternatives.
- ADR includes a section "what changes if we revisit this in 12 months" (e.g., if `oxc` becomes stable, switch).
- The benchmark script is committed under `tools/perf/` (so future maintainers can re-run).

**Review:**
- [ ] Decision is reversible (the parser is wrapped behind one helper module, not spread across checks).
- [ ] Install impact is documented in the README's stack-fit table.

### Story v2.S0.2 — Severity tier definition + `severity_map.yaml`

**Files:**
- `.harness/severity_map.yaml` (new)
- `.harness/schemas/severity_map.schema.json` (new)
- `tests/harness/test_severity_map_completeness.py` (new)

**Severity tiers (four):**

| Tier | Meaning | Pre-commit blocks? | CI blocks? |
|------|---------|--------------------|------------|
| `P0_security` | A vulnerability or compliance gap (Q13.*) | yes | yes |
| `P1_correctness` | Code that will produce wrong results / break in prod (Q7.no-blocking-sleep, Q8.text-call) | yes | yes |
| `P2_quality` | Maintainability / readability (Q15.docstring, Q22) | warn only | yes |
| `P3_style` | Pure stylistic preference (Q18 naming) | warn only | warn only |

**`severity_map.yaml` shape:**

```yaml
schema_version: "1"
rules:
  Q13.route-needs-auth:
    tier: P0_security
    why: |
      Every mutating route must require an authenticated user.
      Missing auth is the #1 cause of "we leaked all our data."
    fix_hint: "add `user: Annotated[User, Depends(<auth_fn>)]` to the handler signature"
  Q7.no-requests:
    tier: P1_correctness
    ...
```

**Acceptance:**
- Every rule across every check has an entry. `test_severity_map_completeness.py` walks `.harness/checks/*.py`, extracts every rule ID, fails if any is missing from the map.
- Schema enforces `tier ∈ {P0_security, P1_correctness, P2_quality, P3_style}`.
- `harness rules` will read this file (Sprint 1 S1.4 dependency).
- New checks added in any future sprint MUST add a severity entry; the test gate enforces it.

**Review:**
- [ ] Every rule ID in the substrate has an entry.
- [ ] Tiers chosen are conservative (default P1; only escalate to P0 if truly security-critical).
- [ ] No empty `why:` or `fix_hint:` blocks.

**Step 2: Implement**

`harness init` orchestrates: detect stack, propose profile, write `.harness/profile.yaml`, run baselines silently, print the green-checkmark summary. The summary is mandatory (~7 lines, ASCII-art-free).

**Edge cases handled (each is a separate code path with its own test):**
- Existing `.harness/` directory → refuse without `--force`.
- Existing pre-commit hook from another tool → leave it alone, print warning, exit 0.
- Read-only filesystem → exit 2 with clear "permission denied" message; NO Python traceback.
- Repo not at git root → detect, exit 2 with "run from the git root".
- Symlinked directories → loop guard in walk-up logic.
- Files with non-UTF-8 bytes during baseline scan → emit `[WARN]` per file, continue.
- Network fail when `--from-git` clones → caught by `subprocess.run(timeout=120)`, clean error.
- GPG key not imported → detected by `harness doctor` post-install; init succeeds with a warning hint.

**Atomic install pattern:**
- Write all new files to `<target>/.harness.staging/` first.
- After all writes succeed, atomically `os.rename(staging, target/.harness)`.
- On any failure, `shutil.rmtree(staging)` and exit cleanly. No partial state.

**Acceptance:**

- All 8 tests pass.
- `harness init --non-interactive` succeeds on a Python repo, a Node repo, a polyglot repo, and an empty directory.
- The empty-directory case prints a `[WARN]` (not error) with "no stack signals detected; using minimal profile" and still installs successfully.
- The first run prints **zero `[ERROR]` lines** even when the consumer has 4000 baselined violations.
- `.harness/profile.yaml` is JSON-Schema-validated by `.harness/schemas/profile.schema.json`.
- Stack detection has its own test fixture set under `tests/harness/fixtures/stack_detector/` (created in S0.3).
- Atomic install: an `init` interrupted by SIGINT leaves no partial state.
- All 8 edge cases above are covered by individual tests.

## Sprint 5 — Multi-repo policy distribution

**Goal:** an organization with 5 repos can roll out a policy change in one commit.

### Story v2.S5.1 — Policy overlay mechanism

```yaml
# .harness/overlays.yaml
schema_version: "1"
overlays:
  - source: github.com/acme/policy-pack
    ref: v2.1.0
    trust_key: <fingerprint>
    precedence: 100  # higher = applied last (wins on conflict)
```

**Conflict resolution (per r1 review):**
- Two overlays declaring the same rule ID → reject at load time with `[ERROR]` listing both sources.
- Per-key policy YAML overrides → highest precedence wins; `harness rules explain` shows which overlay supplied each value.
- `harness overlay list` shows all active overlays + their precedence.

**Trust:** mandatory `--trust-key`. No overlay loads without verified signature. Key rotation supported via `--trust-keys <fpr1,fpr2>` (accept either; document deprecation window).

**Acceptance:**
- `harness overlay add <url> --ref <tag> --trust-key <fpr>` registers.
- Trust pinning mandatory; rejected without.
- Conflict detection tested for both check ID + policy YAML key cases.
- 15 tests: overlay-load, trust-mismatch (rejected), version-pin, removed, key-rotation, conflict-checks, conflict-yamls, precedence-resolution.

## Cross-sprint requirements (apply to every story)

These are review-gate items every story must pass:

### Test coverage

- Every new file in `tools/` ≥ 95% line + branch coverage.
- Every new check ≥ 95%.
- Every new generator ≥ 90%.
- New code in any PR: ≥ 95% on the diff (use `diff-cover`).
- Branch coverage enforced, not just line.

### No silent error handling

The Q17 rule the harness enforces on consumers applies to the harness itself. Every silent swallow needs a `# Q17-EXEMPT: <reason>` comment that `Q17.exempt-needs-reason` (added in S3.2) verifies.

### Clear contracts

Every public function in `tools/`:
- Has a docstring.
- Has type hints.
- Documents the exact return shape.
- Documents what it raises.

Every inter-process boundary is recorded in `docs/CONTRACTS.md`.

### Reliability gate

- No flaky tests. The `flake-zero` CI job (S3.4) runs every test 5x.
- Every subprocess call has a `timeout=` argument.
- Every file write is atomic (`.new` → rename).
- Concurrent writes use `fcntl.LOCK_EX`.

### Maintainability checks

- Every check ≤ 300 lines.
- No metaclasses, no decorators-that-modify-the-class.
- Every YAML has a JSON Schema with `additionalProperties: false`.
- Every new feature has an ADR before it ships.

### Permanent regression coverage

Every closed bug (B/S findings) has a permanent test (S0.4 establishes the floor; new bugs add to it).

### One-core-job discipline

Before merging, ask: does this story serve **the core job** (don't drift, don't burn tokens, don't break trust)? If the connection is more than one hop, defer.

|------|-----------|--------|------------|
