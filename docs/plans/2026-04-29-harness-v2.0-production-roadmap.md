# ai-harness v2.0 — Production Roadmap

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans. **All implementation work happens in `github.com/inder1991/ai-harness` (the standalone source repo).**

**Date:** 2026-04-29
**Audience:** maintainers + contributors planning the v2.0 line.
**Author voice:** Head of Developer Experience.

---

## North star

> **Pick one core job and do it exceptionally well.**

The core job: **make AI-assisted development inside any consumer repo not drift, without burning tokens, without breaking trust.**

Everything in v2.0 is in service of that. Anything that drags us away from it (a web UI, an LLM-judge tier, a sandbox, an evals platform) is explicitly out of scope. The HARNESS_CARD's `not_covered` field is treated as a contract.

## Operating principles for the whole sprint

These are non-negotiable. Every story is reviewed against them before it's marked done.

1. **Reliability over feature velocity.** No story merges until it has tests and a passing CI. A flaky test is a failing test.
2. **Maintainability over cleverness.** A 30-line check that any contributor can read beats a 10-line check with metaclasses.
3. **No silent error handling.** Every failure path either:
   - emits a clearly-formatted error the user can act on, OR
   - exits with a documented non-zero code that the orchestrator surfaces.
   Bare `except: pass` is a bug; `except Exception:` without a logger or re-raise is a bug. The harness already enforces this on consumers (Q17); v2.0 holds itself to the same standard.
3. **Clear contracts between components.** Every internal interface is documented (function signature + docstring + exact return shape). The H-16 emit format is the canonical inter-process contract. Two components never share state through "well, they both happen to use the same Path" — they share via explicit arguments or via a documented file contract.
4. **Test coverage gate.** ≥85% line coverage on `tools/` and `.harness/checks/` measured per release. Cumulative test count grows; never shrinks without an ADR.
5. **Stack profiles, not stack assumptions.** Every check declares which profiles it belongs to in its module docstring frontmatter. The orchestrator only runs checks whose profile is enabled in the consumer's `.harness/profile.yaml`.
6. **First-run delight, not first-run trauma.** A fresh `harness init` on any repo prints a single green-checkmark summary, not a wall of legacy violations.

```python
def test_formatter_groups_by_severity():
    findings = [
        Finding("Q13.route-needs-auth", "backend/api.py:42", "P0_security"),
        Finding("Q15.docstring", "backend/x.py:10", "P3_quality"),
        Finding("Q15.docstring", "backend/y.py:20", "P3_quality"),

    assert "Why:" in out
    assert "Fix:" in out
    assert "More:" in out
    assert "harness rules explain Q13.route-needs-auth" in out

def test_formatter_machine_readable_mode_unchanged():
    """--json mode emits the H-16 contract verbatim — orchestrator stability."""
    out = format_findings([Finding("Q13.route-needs-auth", "x.py:1", "P0")], mode="json")
    parsed = json.loads(out)
    assert parsed["findings"][0]["rule"] == "Q13.route-needs-auth"

def test_formatter_passing_summary():
    out = format_findings([], mode="human")
    assert "✓" in out
    assert "rules passed cleanly" in out
```

**Step 2: Implement**

The formatter takes a list of `Finding` records and emits in one of three modes:
- `human` (default for terminal) — colors, severity groups, collapsed counts, "why/fix/more".
- `json` (machine consumer) — exact H-16 lines, one per finding, plus a JSON summary.
- `raw` (legacy) — exact H-16 lines, no summary. For backward compat with anything that parsed v1.x output.

A `severity_map.yaml` declares each rule's severity tier (P0_security, P1_correctness, P2_quality, P3_style). Loaded once.

Each rule has an explanation entry in `.harness/checks/<check>_explainer.yaml` with `why:` and `fix:` blocks. (For v2.0 we ship default explainers extracted from the existing module docstrings; v2.1+ refines them.)

**Acceptance:**

- All 4 tests pass.
- `harness check` shows colors when `stdout.isatty()`, no colors when piped (use `colorama` or roll a simple ANSI helper).
- Severity ordering: P0 → P1 → P2 → P3; within each, alphabetical by rule ID.
- Findings of the same rule across multiple files collapse to one block with "N occurrences" and `--detail` to expand.
- `harness check --json` emits a stable JSON shape (documented contract).
- `harness check` exit code: 0 on PASS, 1 on any P0+P1 finding, 0 on only P2/P3 findings (with a "X non-blocking warnings" note). This is a behavior change vs v1.x's exit-1-on-any-finding; documented in v2.0 release notes.
- Pre-commit hook calls `harness check --pre-commit` which preserves v1.x behavior (exit 1 on any finding).

**Review checklist:**
- [ ] Backward compat: `make validate-fast` continues to call the H-16-emitting orchestrator; humane formatting is layered on top, not replacing it.
- [ ] Test coverage on `output_formatter.py` ≥ 90%.
- [ ] No silent error handling — if the severity_map.yaml is missing, the formatter falls back to "uncategorized" and emits a single `[WARN]`.
- [ ] Explainer YAMLs are JSON-Schema-validated (typos fail fast).

**Acceptance:**

### Story v2.S3.3 — Deterministic regen guarantee

**Files:** `tools/run_harness_regen.py`, every generator.

Run `make harness` twice in a row on the same code; the diff must be empty. Today this is *probably* true but isn't tested. Add a CI step:

```bash
make harness
git status --short .harness/generated/  # must be empty
make harness
git status --short .harness/generated/  # must still be empty
git diff --exit-code .harness/generated/
```

Any generator that produces non-deterministic output (Python `set` ordering, dict ordering on older Pythons, timestamp injection, absolute paths) fires the gate.

**Acceptance:**
- A new CI job `byte-deterministic-regen` runs the double-build + diff.
- Gate passes on `main`.
- Add `harness doctor` check: "generators are byte-deterministic ✓".

### Story v2.S3.4 — Flake-zero test gate

**Acceptance:**

## Sprint 4 — Polyglot: profile composition + node-pack

**Goal:** Node consumers get a "native" experience equivalent to today's Python-React experience.

### Story v2.S4.1 — Profile composition mechanism

**Files:** `tools/profiles.py`, `.harness/profiles/{python-react,python-only,node-react,node-only,minimal,go-only}.yaml`.

A profile YAML lists which check-packs to enable:

```yaml
# .harness/profiles/python-react.yaml
extends:
  - python-backend
  - react-frontend
  - cross-cutting
  - self-tests
```

Check packs are subdirectories under `.harness/checks/`:
- `.harness/checks/python-backend/` — `backend_async_correctness.py`, `backend_db_layer.py`, etc.
- `.harness/checks/react-frontend/` — `frontend_*.py`

## Sprint 5 — Multi-repo policy distribution

**Goal:** an organization with 5 repos can roll out a policy change in one commit.

### Story v2.S5.1 — Policy overlay mechanism

A consumer can declare:

```
# .harness/overlays.yaml
overlays:
  - source: github.com/acme/policy-pack
    ref: v2.1.0
    trust_key: <fingerprint>
```

The harness pulls the overlay during `harness upgrade`, applies it on top of the base substrate, validates that the overlay's checks coexist with the base. Conflicts (two checks with the same rule ID) fail loudly.

**Acceptance:**
- `harness overlay add <url> --ref <tag> --trust-key <fpr>` registers an overlay.
- Trust pinning is mandatory — no overlay loads without a verified signature.
- Overlay can add new checks, override existing policy YAMLs (with override ADR required), or add new profiles.
- 12 tests covering: overlay-load, overlay-trust-mismatch (rejected), overlay-conflict (rejected), overlay-version-pin, overlay-removed.

### Story v2.S5.2 — Signed pack registry

A maintainer publishes a pack (e.g., `acme/policy-pack`) as a signed git tag. Consumers pull it via `harness overlay add`. The org's checks live in one place; every consumer gets updates atomically.

**Acceptance:**
- Documented release flow for maintainers (similar to v1.x sign_release.sh, scoped to a pack).
- Consumer-side update is `harness upgrade`; rollback is `harness overlay rollback`.

### Story v2.S5.3 — Audit-log replay

A central org dashboard can ingest all consumers' `.harness/.failure-log.jsonl` files (uploaded by CI) and run `harness telemetry --aggregate` over the merged stream.

**Acceptance:**
- `harness telemetry --aggregate <dir>` reads many failure logs and produces an org-wide trend report.
- No PII in the log shape (already true; verified by test).

## Cross-sprint requirements (apply to every story)

These are review-gate items every story must pass:

### Test coverage

- Every new file in `tools/` ≥ 90% line coverage (measured by `coverage.py`).
- Every new check ≥ 95% (the rule logic is pure; no excuse for less).
- A new CI job `coverage-gate` runs after `make validate-full` and fails the PR if coverage drops below the gate.

### No silent error handling

The Q17 rule the harness enforces on consumers applies to the harness itself:
- `except: pass` is forbidden (no exceptions, no comments — just don't).
- `except Exception:` requires either a logger call OR a re-raise in the body.
- Any deliberate exception suppression carries a `# Q17-EXEMPT: <reason>` comment.

### Clear contracts

Every public function in `tools/`:
- Has a docstring.
- Has type hints.
- Documents the exact return shape (especially for functions that return dicts/tuples — name the keys/positions).
- Documents what it raises.

Every inter-process boundary (subprocess invocation, file written to disk, git operation) has a documented contract recorded in `docs/CONTRACTS.md`.

### Reliability gate

- No flaky tests. The `flake-zero` CI job (Sprint 3) runs every test 5x; any flake fails the gate.
- Every subprocess call has a `timeout=` argument (already enforced for git ops since v1.2.0; v2.0 extends this rule to ALL subprocess calls in the substrate).
- Every file write is atomic (`.new` → rename) — already true for baselines; v2.0 audits and extends to every write.

### Maintainability checks

- Every check ≤ 300 lines. If a check grows past that, refactor (extract helpers into `_common.py`).
- No metaclasses, no decorators-that-modify-the-class. The substrate is read by humans; clever code is technical debt.
- Every YAML has a JSON Schema; schema gates are mandatory.
- Every new feature has an ADR under `docs/decisions/` before it ships.

### One-core-job discipline

Before merging, ask: does this story serve **the core job** (don't drift, don't burn tokens, don't break trust)? If the connection is more than one hop, defer.

Examples of correct deferrals:
- A web dashboard for telemetry → defer (terminal `harness telemetry` is enough).
- LLM-assisted rule explanations → defer (canned explainers are enough; LLM costs money).
- A VS Code extension → defer (Claude Code is the integration; an extension is duplicate effort).

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `harness` CLI rollout breaks existing `make` consumers | Low | High | CLI delegates to existing scripts; both surfaces co-exist through v2.x. |
| Profile splitting breaks v1.x consumers on upgrade | Medium | High | Migration script is mandatory part of `harness upgrade`; tested on 5 real consumer repos before v2.0 ships. |
| Node pack's tree-sitter dependency adds heavy install | Medium | Medium | Lazy-load: install only when `node-backend` is in the profile; document the trade-off. |
| Q17 self-audit surfaces large refactor | Medium | Medium | Schedule fix work in Sprint 3 explicitly; don't let it block Sprint 1's CLI release. |
| Auto-fixers introduce bugs that ship as commits | Low | High | Every auto-fixer requires `--apply` confirmation OR `-y` for CI. Diffs always shown. Auto-fixed commits land in feature branches, never directly to main. |
| Multi-repo overlay creates supply-chain attack surface | Medium | Critical | Trust pinning is mandatory; overlays without `trust_key` are rejected. |
